Helix 5 → Helix 6

The short version: Adobe is turning two control desks with different rules into one front door with a more consistent map.

Two buildings become one campus.

Forget implementation detail for a moment. In Helix 5, authoring and site operations live behind different doors. Helix 6 pulls them under one service.

Helix 5 · today

Two control desks

A client must know which building owns which job.

DA Admin
admin.da.live
source files · lists · versions

Helix Admin
admin.hlx.page
preview · publish · status · code

Helix 6 · emerging

One front door

The same service understands content, lifecycle, code, configuration, and jobs.

AEM Admin API
api.aem.live/{org}/sites/{site}/…

Special lane for code
/{owner}/repos/{repo}/code/{ref}/…

Plain English: H6 is mostly a control-plane redesign. It changes how tools talk to Adobe services—not the idea of a webpage itself.

The subtle part: H5 puts main, the code branch, inside most operation URLs. H6 site lifecycle URLs do not. Code still has its own branch-aware route.

The four changes that matter.

If you remember these four cards, you understand most of the migration risk.

Change 01

The front door changes

admin.da.live + admin.hlx.page → api.aem.live

A client must detect which door a particular site uses. The rollout is per-site, not one global switch.

Change 02

The address reads like a resource

/preview/org/site/main/about → /org/sites/site/preview/about

Site identity moves to the front. The branch disappears from lifecycle operations.

Change 03

The message body changes

DA multipart form field → H6 raw body with an explicit content type.

Lists, copy/move, versions, bulk operations, and jobs also have different wire shapes.

Change 04

Branch belongs to code

H5 habit: branch in nearly every admin URL. H6 model: site lifecycle plus a repo/ref code lane.

Content/site operations and Git code operations become two explicit coordinate systems.

Is Helix 6 done?

The honest answer is “real, running, and still integrating.” Separate backend maturity from client migration maturity.

Real today

The service exists

  • Continuously released
  • Selected upgraded sites
  • Live upgrade signal
  • Major API families implemented

Ongoing

Clients are adapting

  • DA Live H6 fixes
  • Sidekick opt-in support
  • Tools compatibility layer
  • End-to-end rollout work

Do not assume

Perfect parity

  • Permission hints still differ
  • No H6 bulk status POST
  • Some documentation is stale
  • Some client routes are transitional

What da-cli should do.

Build a bilingual receptionist. Commands should say “preview this page”; one compatibility layer should decide how to speak H5 or H6.

Stable intent
da-cli commands
source · preview · publish · code · status
New seam
Admin client
detect profile · normalize paths · responses · jobs · errors
Translator A
H5 adapter
preserve current behavior exactly
Translator B
H6 adapter
explicit api.aem.live resource routes
  1. Extract H5 first: move today’s logic without changing behavior.
  2. Detect per site: read the upgrade header and cache carefully.
  3. Normalize results: commands see one stable internal model.
  4. Test both dialects: one behavioral suite, two adapters, two fixture sites.
The safe strategy: do not wait for a universal H6 launch—and do not flip everything at once. Add the translation seam now, keep H5 working, and activate H6 site by site.

Pocket glossary.

These definitions are intentionally non-lawyerly.

Control plane
The APIs and tools that tell the platform what to do. It is not the webpage your visitor sees.
Source
The authored document or data before it becomes a delivered webpage.
Preview
Build or refresh the non-production representation used for review.
Live / publish
Promote prepared content to the public delivery tier.
Site
The H6 lifecycle identity used for content, preview, publish, status, and configuration.
Repo / ref
The Git repository and branch or tag identity. In H6 it remains especially important for code.
Wire format
The exact HTTP method, URL, headers, and body a client sends.
Adapter
A translator that lets one command work against different backend contracts.
Job
A tracked unit of work, such as a large preview or publish operation.
You do not need to learn two products. You need one mental model: H6 unifies the control plane, while a compatibility layer protects clients from the changed grammar.
