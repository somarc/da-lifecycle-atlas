Helix 6

The short version: Adobe is turning two control desks with different rules into one front door with a more consistent map.

What does not changeYour site still has authored content, code, preview, and live delivery.
What does changeWhere tools send requests, how they address a site, and the shape of some messages.
01 · The mental model

Two buildings become one campus.

Forget implementation detail for a moment. In Helix 5, authoring and site operations live behind different doors. Helix 6 pulls them under one service.

Helix 5 · today

Two control desks

A client must know which building owns which job.

DA Adminadmin.da.live
source files · lists · versions
Helix Adminadmin.hlx.page
preview · publish · status · code
→
Helix 6 · emerging

One front door

The same service understands content, lifecycle, code, configuration, and jobs.

AEM Admin APIapi.aem.live
/{org}/sites/{site}/…
Special lane for code/{owner}/repos/{repo}/code/{ref}/…
Plain English: H6 is mostly a control-plane redesign. It changes how tools talk to Adobe services—not the idea of a webpage itself.

03 · The four changes that matter

Same nouns. New grammar.

If you remember these four cards, you understand most of the migration risk.

CHANGE 01

The front door changes

admin.da.live
admin.hlx.page
→
api.aem.live

A client must detect which door a particular site uses. The rollout is per-site, not one global switch.

CHANGE 02

The address reads like a resource

/preview/org/site/main/about
→
/org/sites/site/preview/about

Site identity moves to the front. The branch disappears from lifecycle operations.

CHANGE 03

The message body changes

multipart form
field: data
→
raw body
Content-Type: text/html

Lists, copy/move, versions, bulk operations, and jobs also have different wire shapes.

CHANGE 04

Branch belongs to code

branch in nearly every admin URL
→
site lifecycle
repo/ref code lane

Content/site operations and Git code operations become two explicit coordinate systems.

04 · Why “branch” is confusing

Content and code were never the same thing.

H6 makes the distinction harder for a client to ignore.

Helix 5 habit

One URL formatter often carries org + repo + branch + path everywhere.

previewmain
publishmain
statusmain
codemain

Helix 6 model

The site owns lifecycle. The repository and ref own code.

site previewno ref
site publishno ref
site statusno ref
repo codebranch/ref

06 · Reality check

Is Helix 6 done?

The honest answer is “real, running, and still integrating.” Separate backend maturity from client migration maturity.

REAL TODAY

The service exists

  • Continuously released
  • Selected upgraded sites
  • Live upgrade signal
  • Major API families implemented
ONGOING

Clients are adapting

  • DA Live H6 fixes
  • Sidekick opt-in support
  • Tools compatibility layer
  • E2E rollout work
DO NOT ASSUME

Perfect parity

  • ACL hints still differ
  • No H6 bulk status POST
  • Some docs are stale
  • Some client routes are transitional
07 · What da-cli should do

Build a bilingual receptionist.

Commands should say “preview this page.” One compatibility layer should decide how to speak H5 or H6.

Stable intent

da-cli commands

source · preview · publish · code · status

→
New seam

Admin client

Detect site profile, normalize paths, responses, jobs, and errors.

→
Translator A

H5 adapter

Preserves current behavior exactly.

Translator B

H6 adapter

Uses explicit api.aem.live resource routes.

Extract H5 firstMove today’s logic without changing behavior.
Detect per siteRead the upgrade header; cache carefully.
Normalize resultsCommands see one stable internal model.
Test both dialectsOne behavioral suite, two adapters, two fixture sites.
The safe strategy: do not wait for a universal H6 launch—and do not flip everything at once. Add the translation seam now, keep H5 working, and activate H6 site by site.
